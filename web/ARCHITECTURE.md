# Web Architecture

## Mission
Work as a conservative engineering assistant for this repository.
Prefer minimal, reviewable changes over initiative.

## Required workflow
1. Read relevant files first.
2. Restate the task briefly.
3. Produce a short plan before implementation.
4. Wait for explicit user approval before running commands, editing files, committing, or pushing.
5. Keep changes minimal and local.
6. Run only the relevant validation commands.
7. Summarize exactly what changed.

## Hard constraints
- Do not make incidental refactors.
- Do not rename things unless explicitly requested.
- Do not add incidental classes, wrappers, styling, data, or behavior beyond the requested change.
- Do not change architecture during implementation tasks.
- Do not modify unrelated files.
- Do not infer product decisions from your own preferences.

## Decision protocol
For non-trivial tasks:
- first critique the proposed direction,
- then offer a simpler alternative if one exists,
- then present a minimal implementation plan.

## Repository rules
- /lab/icons shows only canonical icons.
- canonical SVG is the default public preview/copy/download target.
- raw source SVG for /lab/icons may be exposed only inside the selected-icon detail surface as an explicit secondary copy/download/code-view action.
- preserve current architecture boundaries unless explicitly asked to refactor them.

## When to stop and ask
Stop before editing if:
- the task requires changing architecture,
- the requested behavior is ambiguous or requires a product decision,
- the change affects multiple layers unexpectedly,
- the change affects public behavior, content model, SEO, or revalidation unexpectedly,
- a requested behavior conflicts with repository rules.

## UI Layering

This project uses a shadcn-first two-layer UI model:

1. `src/components/ui/*`  
   Literal shadcn component source generated by the official CLI and kept as close to upstream as possible.
2. `src/ui-kit/*`  
   Public app entrypoint and the only allowed layer for shared project-level UI customization, wrappers, defaults, and policy.

Application code must import UI only from the public entrypoint `@/ui-kit`.

Priority order:

1. Use official shadcn components, patterns, and composition first.
2. Use usage-level composition for page layout only.
3. Touch `src/components/ui/*` only when:
   - re-generating from the official shadcn CLI,
   - applying a literal upstream update,
   - or making a narrowly justified primitive-level fix that cannot live anywhere else.
4. Put shared project-level UI behavior and styling policy in `src/ui-kit/*`, not in usage-layer.

## Import Rules

- Allowed in app/features/pages/components:
  - `@/ui-kit`
- Disallowed in app/features/pages/components:
  - `@/components/ui/*`
- Imports from `@/components/ui/*` are allowed only inside:
  - `src/ui-kit/**/*`
  - `src/components/ui/**/*`

This is enforced by ESLint (`no-restricted-imports`) in `eslint.config.mjs`.

## UI Construction Rules

Build interfaces with shadcn structure first.

### Components

1. Prefer the component, pattern, or recipe that shadcn already provides before inventing new markup.
2. When the official docs show a structural pattern, follow that structure literally unless there is a clear repo-specific reason not to.
3. Keep `src/components/ui/*` literal. Do not add project-level tweaks there unless the change is explicitly an upstream-aligned or CLI-driven update.
4. If the project needs shared visual or behavioral customization on top of shadcn, do it in `src/ui-kit/*`.
5. Keep `src/ui-kit/*` structured and minimal:
   - shared defaults,
   - shared wrappers,
   - shared className composition,
   - export curation.
6. Do not turn `src/ui-kit/*` into a second hand-written component library with page-specific behavior.

### Forms

1. Use `Field`, `FieldGroup`, `FieldContent`, `FieldLabel`, `FieldDescription`, `FieldError`, and related shadcn field primitives whenever building form UIs.
2. Do not replace field composition with hand-written `div` + `flex` wrappers when the field primitives already model the structure.
3. Do not override:
   - align-items behavior of a field row,
   - internal rhythm of a field/control row,
   - control row state treatment,
   - micro-spacing already handled by shadcn field primitives.
4. If a project-wide form adjustment is needed, encode it once in `src/ui-kit/*`, not ad hoc in feature code.

### Usage-layer classes

Usage-layer classes are allowed for page/layout composition only:

- grid
- gap between blocks
- width of columns or sections
- section order
- page-level spacing around already-built components

Usage-layer classes must not redefine the visual model of a UI primitive or form pattern.

### Cards and sections

1. Prefer stock shadcn card/header/content/footer composition.
2. Do not add extra wrappers, “section” chrome, or invented panel systems when a `Card`, `FieldGroup`, `Table`, `Alert`, or other existing primitive already models the block.
3. Prefer structure from official shadcn docs over local inventions.
4. If the project needs a shared card or section treatment beyond stock shadcn, put that treatment in `src/ui-kit/*`.

### Reinstallation and updates

1. When resetting or updating shadcn, use the official CLI.
2. For full resets, prefer:
   - `npx shadcn@latest init --preset <code> --force --reinstall`
   - `npx shadcn@latest add --all --overwrite`
3. Treat shadcn docs and CLI output as the source of truth for what is shipped as a component versus what is only documented as a recipe/example.

## Update Workflow
Run the smallest validation set that matches the change.

After any UI changes:

1. `npm run lint`
2. `npm run build`

Before push or when full confidence is needed:

1. `npm run check`
